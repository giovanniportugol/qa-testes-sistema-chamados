# Plano de Testes — Sistema de Chamados

## 1. Objetivo

Este plano de testes tem como objetivo documentar a estratégia de validação de um sistema fictício de chamados, utilizado como projeto acadêmico em desenvolvimento para estudos em QA/Testes de Software.

O foco é praticar a organização de testes manuais, identificação de cenários, documentação de casos de teste, registro de bugs, rastreabilidade dos requisitos e geração de evidências.

---

## 2. Escopo dos Testes

Os testes contemplam funcionalidades básicas de um sistema de chamados, incluindo:

* Abertura de chamados
* Validação de campos obrigatórios
* Listagem de chamados
* Atualização de status
* Conclusão de chamados
* Arquivamento de chamados
* Consulta de chamados
* Organização e registro das evidências

---

## 3. Fora do Escopo

Neste momento, não fazem parte do escopo:

* Testes de desempenho
* Testes de carga
* Testes de segurança avançados
* Testes de acessibilidade
* Testes em múltiplos navegadores
* Testes automatizados completos

Esses itens poderão ser estudados e ampliados futuramente, conforme a evolução do projeto.

---

## 4. Estratégia de Testes

A estratégia adotada para este projeto consiste na execução de testes manuais focados nas principais funcionalidades do sistema de chamados.

Os testes serão realizados com base nos requisitos funcionais definidos, utilizando casos de teste documentados e registro formal dos defeitos encontrados.

A rastreabilidade entre requisitos, casos de teste e bugs será mantida por meio da Matriz de Rastreabilidade do projeto.

---

## 5. Tipos de Teste

Serão considerados os seguintes tipos de teste:

* Testes Funcionais
* Testes Manuais
* Testes Positivos
* Testes Negativos
* Testes de Validação de Campos
* Testes Exploratórios
* Testes de Regressão Simples

---

## 6. Ambiente de Teste

Ambiente utilizado para estudo e execução dos testes:

* Sistema Operacional: Windows 11
* Navegador: Google Chrome
* Editor de Código: Visual Studio Code
* Versionamento: Git e GitHub
* Documentação: Markdown
* Projeto: Sistema Fictício de Chamados

---

## 7. Critérios de Entrada

Os testes poderão ser iniciados quando:

* As funcionalidades básicas estiverem descritas
* Os cenários de teste estiverem definidos
* Os casos de teste estiverem documentados
* O ambiente de teste estiver disponível
* O repositório estiver organizado para registro das evidências

---

## 8. Critérios de Saída

Os testes poderão ser considerados concluídos quando:

* Os principais casos de teste forem executados
* Os resultados forem registrados
* Os bugs encontrados forem documentados
* As evidências forem organizadas
* A matriz de rastreabilidade estiver atualizada

---

## 9. Critérios de Aceitação

Uma funcionalidade será considerada aprovada quando:

* Executar o comportamento esperado
* Validar corretamente os campos obrigatórios
* Apresentar mensagens claras ao usuário
* Cadastrar ou atualizar informações corretamente
* Não apresentar erros críticos durante a utilização
